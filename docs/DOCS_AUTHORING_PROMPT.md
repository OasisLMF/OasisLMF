# Prompt: document a new OasisLMF feature

A reusable prompt for documenting a new feature in the OasisLMF in-repo docs
(`docs/`). Fill in the **Feature context** block and hand the whole thing to
Claude Code (or follow it yourself). It encodes the conventions from
`DOCS_STRATEGY.md` so features get documented in the right place, the right way,
and in the same PR as the code.

---

## Feature context — FILL THIS IN

- **Feature name:** _<short name>_
- **What it does / why it exists:** _<1–3 sentences>_
- **Code it lives in:** _<module path(s), e.g. `oasislmf/pytools/<pkg>`, or a CLI
  command, or an `analysis_settings` / `model_settings` field>_
- **User-facing surface:** _<new CLI flag? config option? new output? new
  behaviour of an existing command? library API?>_
- **Who uses it:** _<analyst / model developer / platform operator / contributor>_
- **Conceptual depth:** _<does it need a "why/how it works" explanation, or is it
  self-explanatory once you know the option exists?>_
- **Related existing pages:** _<any pages that should link to or from this>_

---

## Task

Document the feature above in the OasisLMF in-repo Sphinx docs. Follow the rules
below exactly.

### 1. Decide which Diátaxis mode(s) the feature needs

Do **not** put everything on one page. Pick only the modes that apply:

- **How-to guide** — almost always. A task-focused recipe: "how to <do the thing
  the feature enables>". Task-oriented, imperative, minimal theory.
- **Reference** — usually **auto-generated**, so you mostly don't hand-write it:
  - Python API → generated by `sphinx-autoapi` from docstrings. Your job is to
    write good **Google-style docstrings** in the code (types + descriptions).
  - CLI options → generated automatically (see `conf.py` `list_options()` →
    `generated_options.rst`). A new `oasislmf` CLI flag appears with no manual
    page; just write a clear `help=` string.
  - `analysis_settings` / `model_settings` fields → update the JSON schema.
- **Explanation** — only if there is real conceptual depth (an algorithm, a
  model, a non-obvious design). Understanding-oriented. **Co-locate it with the
  code** it describes.
- **Tutorial** — only if the feature warrants a full guided, learn-by-doing
  walkthrough for newcomers. Most features do not need one.

State your choice and why before writing.

### 2. Put files in the right place (co-location)

Current layout under `docs/source/`:

```
index.rst              landing page: audience cards + per-section toctrees
tutorials/             learning-oriented (index.md + pages)
reference/index.md     config, env vars, and the auto-generated Python API
reference/api/         GENERATED by autoapi — never edit by hand
explanation/           understanding-oriented, co-located with the code
  financial-module.rst, fm-architecture.md, index.md
_static/images/        figures
installation.rst, building-and-running-models.rst, logging-configuration.rst
                       existing how-to pages (currently at source root, grouped
                       under the "How-to guides" toctree caption in index.rst)
```

Rules:
- **New how-to page:** create `docs/source/how-to/<feature>.md` (prefer a
  `how-to/` folder going forward) and add it to the "How-to guides" toctree in
  `index.rst`. If you keep it at source root to match the existing pages, still
  add it to that toctree.
- **New explanation page:** `docs/source/explanation/<feature>.md`, added to the
  toctree in `explanation/index.md`. Explanation lives here **because it is
  co-located with the algorithm** — the whole point is that a behaviour change
  and its explanation move together in one PR.
- **Reference:** don't write API pages by hand. If the feature is in a pytools
  subpackage **not yet covered** by autoapi, widen `autoapi_dirs` in `conf.py` to
  include it (keep it scoped per subsystem — do NOT revert to a whole-package
  dump).
- **Images:** put them in `docs/source/_static/images/<area>/` and reference with
  an absolute path — `/_static/images/<area>/<file>` (rst `.. image::`) or
  `![alt](/_static/images/<area>/<file>)` (MyST).

### 3. Authoring conventions

- **Prefer MyST Markdown (`.md`)** for new pages. Existing `.rst` is fine to
  leave; both compile.
- **Cross-link the modes** for the feature: the how-to links to the explanation
  ("why this works, see …") and to the reference (the exact options); the
  explanation links back to the reference. This is the payoff of the split.
- Use `sphinx-copybutton`-friendly fenced code blocks with a language, e.g.
  ```` ```bash ````.
- Match the tone and depth of the neighbouring pages.

### 4. Gotchas — do NOT trip these (see `DOCS_STRATEGY.md` §9)

- **Never use the `.. contents::` directive.** Furo already renders an "On this
  page" TOC in the right sidebar and turns `.. contents::` into a red, reader-
  visible ERROR box — and this does **not** show up as a build warning.
- **Don't hardcode external links to `github.com/OasisLMF/ktools/...`** — that
  content is being migrated into OasisLMF; use an in-repo cross-reference if the
  target exists, otherwise a `:doc:` / `{doc}` link, and flag it if it doesn't.
- Keep autoapi **scoped per subsystem**; never point it at the whole package.

### 5. Build and verify (a clean build is NOT enough)

From the `docs/` directory:

```bash
# build (use your local venv, e.g. ~/venv_3.12/bin/sphinx-build)
sphinx-build -b html source build/html

# 1. it must exit 0 with no NEW warnings for your pages
# 2. it must RENDER clean — grep the built HTML for the Furo error banner:
grep -rl "does not work well with existing styling" build/html/ && echo "FURO ERROR BOX PRESENT" || echo "render clean"

# 3. no dead links you introduced:
sphinx-build -b linkcheck source build/linkcheck
```

Fix anything your pages caused. (Pre-existing `fm` docstring warnings are known;
don't chase those unless your feature is in those modules.)

### 6. Ship it in the same PR as the code

The docs change goes in **the same pull request** as the feature. If the repo has
a "docs updated / not needed" PR checklist item, tick it.

---

## Output expected from the doc author

1. A one-line statement of which Diátaxis mode(s) you chose and why.
2. The new/changed files with their paths.
3. Any `conf.py` change (e.g. widening `autoapi_dirs`) with justification.
4. The verification results (build exit code, render-clean grep, linkcheck).
